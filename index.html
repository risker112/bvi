KAP 01

Prehľad bezpečnosti počítačových systémov

CIA Triáda

NIST definuje bezpečnosť počítačov ako ochranu automatizovaného informačného systému s cieľom zachovať integritu, dostupnosť a dôvernosť zdrojov (hardware, software, firmware, dáta, telekomunikácie).

C
Dôvernosť
Zabezpečenie, že informácie nie sú sprístupnené neoprávneným osobám. Zahŕňa aj súkromie (privacy).
I
Integrita
Zabezpečenie, že informácie a programy sú menené len oprávneným spôsobom. Zahŕňa integritu dát aj systému.
A
Dostupnosť
Systémy fungujú promptne a služba nie je odmietnutá oprávneným používateľom.

CIA Triáda (Wikimedia Commons)

Rozšírenie CIA: Niektorí odborníci pridávajú Autentickosť (možnosť overiť pravosť) a Zodpovednosť (akcie entít sa dajú sledovať a priradiť).

Základná terminológia (RFC 2828)

Hrozba (Threat)
Potenciálne porušenie bezpečnosti — okolnosť, schopnosť alebo udalosť, ktorá by mohla spôsobiť škodu.
Zraniteľnosť (Vulnerability)
Slabina v návrhu, implementácii alebo správe systému, ktorá môže byť zneužitá.
Útok (Attack)
Cielený pokus narušiť bezpečnostnú politiku — realizácia hrozby.
Protiopatrenie (Countermeasure)
Akcia, zariadenie alebo technika, ktorá znižuje hrozbu alebo zraniteľnosť.
Riziko (Risk)
Očakávaná strata vyjadrená pravdepodobnosťou, že hrozba zneužije zraniteľnosť.
Bezpečnostná politika
Sada pravidiel a praktík, ktoré špecifikujú ako systém poskytuje bezpečnostné služby.

Bezpečnostná stratégia (3 aspekty)

1
Špecifikácia/politika: Čo má bezpečnostná schéma robiť?
2
Implementácia/mechanizmy: Prevencia, detekcia, reakcia, obnova.
3
Správnosť/zaistenie: Naozaj to funguje? — stupeň dôvery, nie formálny dôkaz.
Skutočné príklady narušení CIA
Equifax 2017 → Dôvernosť
Neaplikovaný Apache Struts patch (CVE-2017-5638) 2 mesiace po vydaní. Útočník získal PII 147 miliónov ľudí. Príčina: zlý patch management.
SWIFT bankový útok 2016 → Integrita
Útočníci vložili podvrhnuté SWIFT platobné príkazy a zakryli záznamy transakcií. Ukradli 81 miliónov USD. Integrita dát narušená.
Mirai botnet DDoS 2016 → Dostupnosť
600 000+ IoT zariadení (kamery, routery) s default heslami infikovaných. DDoS na Dyn DNS → polovica internetu nedostupná hodiny.
NotPetya 2017 → Integrita + Dostupnosť
Malware maskovaný ako ransomware permanentne prepisoval MBR diskov. Škody 10 mld. USD. Maersk musel preinštalovať 45 000 PC.

CVSS skóre → závažnosť

Taxonómia bezpečnostných kontrol
Preventívne (Preventive)
Bránia útoku skôr, než nastane. Príklady: firewall, ACL, šifrovanie, MFA, patches, security training.
Detektívne (Detective)
Zisťujú prebiehajúci alebo hotový útok. Príklady: IDS/IPS, antivírus, audit logy, SIEM, honeypot.
Korektívne (Corrective)
Obnovujú stav po útoku. Príklady: zálohy, incident response plán, patching po exploite, rebuild systému.
Odstrašujúce (Deterrent)
Odrádzajú potenciálneho útočníka. Príklady: právne hrozby, security kamery, warning bannery, audit oznámenia.
Životný cyklus incidentu — ako útok prebieha

Každá fáza je príležitosť pre obrancov zastaviť útok. Ak útočník zlyhá kdekoľvek v reťazci, útok sa zastaví — preto "defense in depth" funguje.

Aký je rozdiel medzi pasívnym a aktívnym útokom?
Pasívny útok (odpočúvanie, sledovanie) neovplyvňuje zdroje — ťažko detekovateľný, dá sa mu predísť šifrovaním. Aktívny útok (MitM, replay, DoS) modifikuje dátový tok — detekovateľný, ale ťažšie mu predísť.
Otázka 4
Čo je to 'vulnerability' (zraniteľnosť)?
RFC 2828: Vulnerability = slabina v návrhu, implementácii alebo správe systému. Threat (hrozba) = potenciálne porušenie bezpečnosti. Attack = realizácia hrozby.
Otázka 5
Akciu bezpečnostnej kamery a warning bannera ('Unauthorized access prohibited') môžeme klasifikovať ako ktorý typ bezpečnostnej kontroly?
Deterrent kontroly odrádzajú potenciálneho útočníka — kamery, legal warnings, bannery. Nezabraňujú útoku priamo, ale znižujú motiváciu útočníka.
KAP 02

Kryptografické nástroje

Symetrická šifra

Klasická metóda — jeden kľúč na šifrovanie aj dešifrovanie. Pred verejno-kľúčovou kryptografiou jediný spôsob. Stále najrozšírenejší typ.

5 komponentov: Plaintext · Šifrovací algoritmus · Tajný kľúč · Ciphertext · Dešifrovací algoritmus

Dva typy útoku: kryptoanalýza (exploatácia slabín algoritmu) vs. brute-force (vyskúšanie všetkých kľúčov).

AlgoritmusKľúčBlokPoznámka
DES56 bit64 bitPrelomený 1998 (EFF, <3 dni, <$250k). Zastaraný!
3DES112/168 bit64 bitDES 3× → pomalý, ale silný. Zahrnutý do FIPS 46-3 (1999).
AES (Rijndael)128/192/256 bit128 bitFIPS 197 (2001). Štandard dnes. 128-bit kľúč — prakticky neprelomiteľný brute-force.
Módy operácie: ECB (jednoduchý, nebezpečný pre dlhé správy) → CBC, CFB, OFB, CTR a ďalšie (riešia slabiny ECB). Stream ciphery sú rýchlejšie, blokové šifry lepšie pre súbory/DB.

Hash funkcie & Autentifikácia správ

MAC (Message Auth. Code)
Zdieľaný tajný kľúč + správa → MAC. Overenie integrity aj zdroja. Príjemca prepočíta MAC a porovná.
Hash funkcie (SHA)
Premenlivá vstup → fixná výstup (digest). Jednosmerná, bezkolízna. SHA-1 (160b, dnes slabý), SHA-256/384/512 (FIPS 180-2).
Vlastnosti silnej hash funkcie: preimage-resistant (2ⁿ) · second preimage-resistant (2ⁿ) · collision-resistant (2ⁿ/²). Heslo v /etc/shadow je hash hesla — nie heslo samotné.

Asymetrická (verejno-kľúčová) kryptografia

Diffie & Hellman 1976 — revolučný prínos. Dva kľúče: verejný (každý ho pozná) a súkromný (vlastník tajný).

  • RSA (1977, Rivest-Shamir-Adleman) — najrozšírenejší. Blokový cipher. Bezpečnosť na faktorizácii. Min. 1024-bit kľúč.
  • Diffie-Hellman — výmena kľúčov (nie šifrovanie), diskrétne logaritmy.
  • DSA/DSS — len digitálny podpis (FIPS 186), nie šifrovanie.
  • ECC — eliptické krivky, rovnaká bezpečnosť pri kratšom kľúči → vhodné pre mobilné zariadenia.

Digitálny podpis

1
Bob vypočíta hash správy (SHA-512)
2
Zašifruje hash svojím súkromným kľúčom → digitálny podpis
3
Pošle správu + podpis Alici
4
Alice dešifruje podpis verejným kľúčom Boba, porovná s hashom správy → overenie
Podpis nezabezpečuje dôvernosť — správa je čitateľná. Zabezpečuje len autenticitu a integritu.
Certifikáty (X.509)
Verejný kľúč + ID vlastníka, podpísané dôveryhodnou treťou stranou (CA — Certificate Authority). Štandard X.509 v3 — používaný v IPsec, TLS, SSH, S/MIME.
ECB vs CBC — "Penguin Problem"

ECB (Electronic Codebook) šifruje každý 128-bitový blok nezávisle. Rovnaký plaintext blok → rovnaký ciphertext blok. Vzory v dátach sú viditeľné!

Legendárny príklad: bitmapový obrázok tučniaka Tux zašifrovaný ECB módou stále ukazuje obrysy tučniaka, pretože rovnaké farebné bloky majú rovnaký ciphertext. CBC to eliminuje reťazením blokov.

AES vnútorná štruktúra (128-bit = 10 kôl)
SubBytes (S-box)
Každý bajt stavu sa nahradí hodnotou z precomputed S-box tabuľky (nelineárna substitúcia). Odolnosť voči lineárnej a diferenciálnej kryptoanalýze.
ShiftRows
Riadky 4×4 stavovej matice sa rotujú o 0, 1, 2, 3 pozície. Zabezpečuje difúziu — rozmiestňuje bajty naprieč stĺpcami.
MixColumns
Každý stĺpec sa násobí fixnou maticou v GF(2⁸). Najsilnejšia difúzna vrstva — každý výstupný bajt závisí od všetkých 4 vstupných bajtov stĺpca.
AddRoundKey
XOR stavu s 128-bitovým kľúčom pre dané kolo (z key schedule). Jediná operácia, kde sa aplikuje tajný kľúč — "kľúčová vrstva".

Kolo 10 vynecháva MixColumns (nepridáva bezpečnosť v poslednom kole).

RSA generovanie kľúčov krok za krokom
1
Vybrať dve veľké prvočísla p a q (napr. 1024-bitové každé)
2
Vypočítať modulus n = p × q
3
Vypočítať Eulerovu totientu: φ(n) = (p−1)(q−1)
4
Vybrať verejný exponent e: 1 < e < φ(n), gcd(e, φ(n)) = 1. Typicky e = 65537
5
Vypočítať súkromný exponent d: d × e ≡ 1 (mod φ(n)) — rozšírený Euklidov algoritmus
6
Public key (e, n) — zverejniť.  Private key (d, n) — uchovať v tajnosti
Šifrovanie: C = Mᵉ mod n   |   Dešifrovanie: M = Cᵈ mod n

Bezpečnosť: faktorizácia n späť na p a q je výpočtovo neuskutočniteľná pri dostatočne veľkom n (≥2048 bit dnes).

TLS Handshake — kryptografia v praxi (HTTPS)
1
ClientHello: Prehliadač pošle podporované TLS verzie, cipher suites, náhodné číslo
2
ServerHello + Certificate: Server vyberie cipher suite, pošle X.509 certifikát (verejný kľúč)
3
Key Exchange: Diffie-Hellman výmena → obaja odvodia rovnaký pre-master secret bez odoslania cez sieť
4
Session Keys: Z pre-master secret sa odvodia symetrické kľúče pre AES šifrovanie a MAC
5
Finished: Obe strany potvrdia úspešnú výmenu → všetka ďalšia komunikácia je šifrovaná AES

TLS kombinuje asymetrickú kryptografiu (RSA/DH) pre bezpečnú výmenu kľúčov a symetrickú (AES) pre rýchle šifrovanie dát — to isté ako PGP.

0/5 otázok
Otázka 1
Prečo bol DES (56-bitový kľúč) vyhlásený za zastaraný?
V roku 1998 EFF (Electronic Frontier Foundation) postavila stroj za ~$250k, ktorý prelomil DES za menej ako 3 dni. 56-bitový kľúčový priestor (2⁵⁶) je príliš malý na brute-force ochranu.
Otázka 2
Čo je 'ECB penguin problem' a prečo je ECB mód nebezpečný?
V ECB móde každý 128-bitový blok sa šifruje nezávisle rovnakým kľúčom. Rovnaký plaintext = rovnaký ciphertext. Obrázok tučniaka Tux zašifrovaný ECB stále odhaľuje obrysy — štatistické vzory sú zachované.
Otázka 3
Aký je účel 'soli' (salt) pri hashovaní hesiel?
Bez soli: všetky účty s heslom 'password123' majú identický hash → jeden rainbow table stačí. So soľou: každý hash je unikátny → útočník musí precomputovať rainbow table pre každý salt zvlášť — prakticky neuskutočniteľné.
Otázka 4
RSA šifrovanie: ako funguje dešifrovanie správy?
RSA: šifrovanie verejným kľúčom príjemcu → dešifrovanie súkromným kľúčom príjemcu. Vzorec: C = Mᵉ mod n (šifrovanie), M = Cᵈ mod n (dešifrovanie). Len príjemca má d (privátny kľúč).
Otázka 5
Čo zaručuje digitálny podpis (a čo NIE)?
Digitálny podpis = hash správy zašifrovaný súkromným kľúčom odosielateľa. Overuje: kto to poslal (autentickosť) a či správa nebola zmenená (integrita). Samotná správa je stále čitateľná → žiadna dôvernosť.
KAP 03

Autentifikácia používateľa

Autentifikácia je základný stavebný blok bezpečnosti — základ pre riadenie prístupu a zodpovednosť. RFC 2828: "Proces overenia identity nárokovanej entitou."

Proces autentifikácie

1
Identifikačný krok: Predloženie identifikátora bezpečnostnému systému (login ID). Identifikátory musia byť prideľované opatrne.
2
Verifikačný krok: Predloženie autentifikačnej informácie, ktorá potvrdzuje väzbu medzi entitou a identifikátorom.

Štyri faktory autentifikácie

Vedomosť Niečo, čo vieš
Heslo, PIN, odpovede na bezpečnostné otázky.
Vlastníctvo Niečo, čo máš
Smartkarta, hardvérový token, fyzický kľúč — "token authenticator".
Existencia Niečo, čím si (statická biometria)
Odtlačok prsta, sietnica, tvár.
Správanie Niečo, čo robíš (dynamická biometria)
Vzor hlasu, rukopis, rytmus písania.

Slabiny hesiel

ÚtokPopis

Specific account attackCielené uhádnutie hesla konkrétneho účtu.

Viacfaktorová autentifikácia (MFA / 2FA)

Cieľ: odstrániť slabiny čisto heslových schém. Vyžaduje kombináciu aspoň 2 faktorov. V Linuxe realizovateľné cez PAM (Pluggable Authentication Module).

TOTP (Time-Based One-Time Password)

  • Zdieľaný tajný kľúč medzi používateľom a serverom (QR kód)
  • Generátor (Google Authenticator, Authy) produkuje nový kód každých 30 sekúnd
  • Kód je neplatný po vypršaní → ochrana aj pri kompromitácii
/etc/shadow formát a hashování s saltom
username:$6$salt$hash:lastchange:min:max:warn:inactive:expire:
root:$6$abc123xyz$dLongHashHere...:19000:0:99999:7:::
Identifikátory algoritmu
$1$ = MD5 (zastaraný!)
$5$ = SHA-256
$6$ = SHA-512 (odporúčaný)
$2y$ = bcrypt (najlepší pre heslá)
Prečo salt?
Bez soli: všetky účty s heslom "password" majú rovnaký hash → rainbow table útok funguje.
So soľou: každý hash je unikátny aj pre rovnaké heslo → rainbow table musí byť precomputed pre každý salt = neuskutočniteľné.
TOTP algoritmus krok za krokom (RFC 6238)
1
Zdieľaný tajný kľúč K (Base32 encoded, naskenovaný ako QR kód)
2
Vypočítaj časový krok: T = ⌊unix_timestamp / 30⌋ (každých 30s sa T zmení)
3
Vypočítaj HMAC-SHA1: hmac = HMAC-SHA1(K, T_as_8bytes_big_endian) → 20 bajtov
4
Dynamic truncation: offset = hmac[19] & 0x0F, berieme 4 bajty od offsetu
5
6-miestny kód: code = (4_bytes & 0x7FFFFFFF) % 1 000 000
Passkeys — budúcnosť autentifikácie (FIDO2 / WebAuthn)
Čo je Passkey?
Kryptografický pár kľúčov uložený v zariadení (Secure Enclave na iPhone, TPM na Windows). Súkromný kľúč nikdy neopustí zariadenie. Webová stránka dostane len podpis.
Výhody oproti heslám
Phishing-resistant: útočník nemôže zachytiť kód, lebo žiadny neexistuje. Bez server-side úlohu hodnôt. Funguje biometriou (FaceID, odtlačok prsta) na strane zariadenia.
Passkeys (Google, Apple, Microsoft od 2022–2023) sú najvýznamnejší posun v autentifikácii za posledných 20 rokov. TOTP/SMS 2FA sú kroky vpred, Passkeys sú skok.

Otázka 1
Smartkarta a hardvérový token patria do ktorého faktora autentifikácie?

Otázka 3
Čo je 'offline dictionary attack' na heslá?
Offline dictionary attack: útočník najprv exfiltruje súbor s hashmi (/etc/shadow), potom lokálne (offline) porovnáva hashe bežných hesiel zo slovníka. Rýchly, lebo nie je limitovaný sieťovými oneskoreniami ani lockoutom.
Otázka 4
Identifikátor $6$ na začiatku záznamu v /etc/shadow indikuje:
Identifikátory v /etc/shadow: $1$ = MD5 (zastaraný!), $5$ = SHA-256, $6$ = SHA-512 (odporúčaný na Linuxe), $2y$ = bcrypt (najlepší pre heslá, ale menej bežný na Linux systémoch).
Otázka 5
Passkeys (FIDO2/WebAuthn) sú odolné voči phishingu, pretože:
Passkeys = asymetrický kryptografický pár. Súkromný kľúč je v Secure Enclave/TPM a nikdy sa neodošle. Podpis je viazaný na origin (doménu) — podvrhnutá stránka dostane podpis pre iný origin → phishing nefunguje.
KAP 04

Riadenie prístupu (Access Control)

X.800: "Predchádzanie neoprávnenému použitiu zdroja, vrátane predchádzania použitia zdroja neoprávneným spôsobom."

Kontext prístupu

Triáda: Autentifikácia → Autorizácia → Audit
Autentifikácia overí identitu → Autorizácia určí povolené akcie → Audit zaznamenáva prístupy pre neskoršiu analýzu.

Základné elementy riadenia prístupu

Subjekt
Entita schopná pristupovať k objektom. Typicky proces reprezentujúci používateľa alebo aplikáciu. Triedy: Owner, Group, World.
Objekt
Zdroj, ku ktorému je prístup kontrolovaný — súbory, záznamy, procesy, zariadenia, sieťové uzly.
Prístupové právo
Read, Write, Execute, Delete, Create, Search — spôsob, akým subjekt môže pristupovať k objektu.

Politiky riadenia prístupu

Prístupová matica (Access Matrix)

Lampson (1969): dvojrozmerná matica — subjekty (riadky) × objekty (stĺpce). Každá bunka = množina prístupových práv.

ACL — Access Control List
Dekompozícia podľa stĺpcov. Pre každý objekt: zoznam subjektov a ich práv. Vhodné na "kto má prístup k tomuto súboru?"
Capability Tickets
Dekompozícia podľa riadkov. Pre každý subjekt: zoznam objektov a povolených operácií. Vhodné na "čo môže tento používateľ?"

Princíp minimálnych privilégií (Least Privilege)

Každej entite sa prideľujú len minimálne potrebné zdroje a oprávnenia na vykonanie jej funkcie. Obmedzuje škody pri chybách, náhodách alebo kompromitácii.
Bell-LaPadula model — Dôvernosť (MAC)

Formálny model pre vojenské systémy. Každý subjekt aj objekt má bezpečnostný štítok: Top Secret > Secret > Confidential > Unclassified.

Simple Security (no read up)
Subjekt nemôže čítať objekt s vyšším štítkom. Vojak (Secret) nemôže čítať Top Secret dokumenty.

Princíp Informácie tečú len nadol — z vyššej do nižšej klasifikácie.
Star Property (no write down)
Subjekt nemôže písať do objektu s nižším štítkom. Zamestnanec s Top Secret clearance nemôže kopírovať info do Unclassified systému.

Princíp Zabraňuje úniku tajomstiev do menej chránených systémov.
Biba model — Integrita (opak Bell-LaPadula)

Kde Bell-LaPadula chráni dôvernosť, Biba chráni integritu. Úrovne: High Integrity > Medium > Low.

No write up (Integrita)
Subjekt nemôže zapisovať do objektu s vyšším stupňom integrity. Nedôveryhodný subjekt nemôže kontaminovať dôveryhodné dáta.
No read down (Integrita)
Subjekt nemôže čítať z objektu s nižším stupňom integrity. Dôveryhodný program sa nepokazia čítaním nedôveryhodných vstupov.
V praxi sa Bell-LaPadula a Biba nedajú súčasne úplne splniť — obchodné systémy ich kombinujú podľa potrieb. SELinux implementuje hybridný model (MAC + RBAC).
Linux oprávnenia — číselný zápis
chmod 755 súbor
# 7 = rwx (vlastník)
# 5 = r-x (skupina)
# 5 = r-x (ostatní)

# Binárna logika:
# r = 4 (100), w = 2 (010), x = 1 (001)
# 7 = 4+2+1 = rwx
# 5 = 4+0+1 = r-x
# 6 = 4+2+0 = rw-

# SUID bit (Set User ID) — spustí sa s oprávneniami vlastníka
chmod u+s /usr/bin/passwd  # passwd beží ako root, aj keď ho spustí bežný user

SUID je príklad Least Privilege v praxi: passwd potrebuje zápisové práva do /etc/shadow, ale bežný user nie. SUID umožňuje presne toto konkrétne zvýšenie oprávnení bez obecného root prístupu.

Čo je hlavný rozdiel medzi DAC a MAC?
DAC (Discretionary): vlastník rozhoduje kto má prístup — môže ho delegovať. MAC (Mandatory): prístup určujú štítky citlivosti (Top Secret, Secret...) a clearance — entita nemôže prístup prenášať.
Otázka 2
Bell-LaPadula 'Star Property' (★) pravidlo hovorí:
Bell-LaPadula má 2 pravidlá: Simple Security (no read up) a Star Property (no write down). ★ zabraňuje úniku tajomstiev — agent s Top Secret clearance nemôže zapisovať do Unclassified systému.
Otázka 3
ACL (Access Control List) je dekompozícia prístupovej matice podľa:
Prístupová matica: subjekty = riadky, objekty = stĺpce. ACL = dekompozícia podľa stĺpcov — pre každý objekt: zoznam subjektov a práv. Capability Tickets = dekompozícia podľa riadkov — pre každý subjekt: čo môže.
Otázka 4
chmod 640 nastaví práva:
6 = rw- (4+2+0), 4 = r-- (4+0+0), 0 = --- (nič). Takže chmod 640 → rw-r-----: vlastník čítaj a píš, skupina len čítaj, ostatní vôbec nič.
Otázka 5
Princíp Least Privilege v praxi znamená:
Least Privilege = minimalizácia potenciálnej škody. Ak je entita kompromitovaná, útočník má len obmedzené oprávnenia. Príklad: webový server potrebuje čítať súbory, nie spúšťať príkazy ako root.
KAP 05

Bezpečnostná analýza & Threat Modeling (Microsoft SDL)

Microsoft SDL — Security Development Lifecycle

Povinná politika v Microsofte od 2004. Bezpečný softvér vyžaduje: vedenie, vzdelávanie, technológie & procesy, zodpovednosť a priebežné zlepšovanie (6-mesačný cyklus).

SDL Princípy bezpečného návrhu

Attack Surface Reduction
Minimalizovať počet exponovaných vstupných bodov. On-by-default → Off-by-default. Autentikovaný prístup > anonymný. Lokálna dostupnosť > internetová.
Defense in Depth
Predpokladaj, že každá vrstva ochrany zlyhá. Ak prenikne firewall, čo ďalšie bráni systém? Viacvrstvová ochrana.
Least Privilege
Predpokladaj, že aplikácia bude kompromitovaná. Potencionálna škoda musí byť minimalizovaná obmedzením oprávnení.
Secure Defaults
Nasaď aplikácie v bezpečnejšej konfigurácii od začiatku. Zákazník dostane bezpečnú skúsenosť "z krabice".
Threat Modeling
Proces pochopenia hrozieb. Hrozby ≠ Zraniteľnosti. Hrozby = čo môže útočník skúsiť; Zraniteľnosti = konkrétny spôsob, ako je to exploitovateľné.
Basic Privacy
Privacy = Empowerment používateľov kontrolovať zbieranie/použitie ich dát. Security + Privacy = dôveryhodná aplikácia.

Validácia: Overenie modelu, enumerovaných hrozieb, mitigácií a predpokladov.

KAP 06

Škodlivý softvér (Malware)

NIST: "Program vložený do systému, zvyčajne skryte, s úmyslom kompromitovať dôvernosť, integritu alebo dostupnosť."

Klasifikácia Malware

Propagácia Ako sa šíri
  • Vírusy — infikujú existujúci spustiteľný obsah
  • Červy (Worms) — exploitujú sieťové zraniteľnosti, samostatné
  • Trójske kone — sociálne inžinierstvo, skrytá funkcionalita
Payload Čo vykoná
  • Poškodenie systému/dát
  • Botnet (zombie agent)
  • Krádež info (keylogging, spyware)
  • Stealthing — skrývanie prítomnosti
  • Ransomware

Sieťové červy (Worms) — rozšírenie

Červ vs. Vírus
Kľúčový rozdiel: červy sa šíria autonómne cez sieť bez potreby infikovať súbory. Exploitujú sieťové zraniteľnosti (napr. buffer overflow v sieťovej službe). Príklady: Morris Worm (1988 — prvý červ, 6000+ UNIX systémov), WannaCry (2017 — EternalBlue exploit SMBv1, 200 000+ počítačov v 150 krajinách).
Propagácia červa
1. Skenovanie siete (náhodné IP)
2. Exploit zraniteľnosti
3. Inštalácia na novom hoste
4. Pokračovanie skenovania
Bot / Botnet
Infikovaný počítač (zombie) čakajúci na príkazy C2 servera (Command & Control). Použitie: DDoS, spam, kryptomining, proxy.
Trójsky kôň
Vyzerá ako legitímny program (hra, utility). Skrýva zákerné funkcie. Sociálne inžinierstvo — obeť sama ho spustí. Na rozdiel od vírusu sa sám nešíri.

Fázy vírusu

1
Dormant (Spánok): Vírus je nečinný, čaká na spúšťač (dátum, kapacita disku, prítomnosť súboru). Nie všetky vírusy majú túto fázu.
2
Propagation (Šírenie): Kópie sa vkladajú do ďalších programov alebo diskových oblastí. Kópia sa môže mutovať (evade detection).
3
Triggering (Aktivácia): Podmienka spustí payload (počet infekcií, dátum, udalosť v systéme). Logic bomb.
4
Execution (Vykonanie): Payload — môže byť neškodný (správa) alebo deštruktívny (mazanie súborov).

Typy vírusov — podľa cieľa

TypPopis

File infectorInfikuje spustiteľné súbory (EXE, COM).
Makro vírusInfikuje skriptovací kód v dokumentoch (Word, Excel, PDF). Platformovo nezávislý!

Techniky ukrytia — evasion

Encrypted virus
Telo vírusu je šifrované náhodným kľúčom, kľúč sa uloží s vírusom. Každá inštancia má iný bitový vzor.
Stealth virus
Skrýva zmeny pred antivírusom (falšuje volania OS). Používa rootkit techniky.
Polymorphic virus
Mutuje pri každej infekcii (náhodné inštrukcie) — podpis sa mení, detekcia podľa signatúry nefunguje.
Metamorphic virus
Kompletne sa prepíše pri každej infekcii. Mení aj správanie, nielen vzhľad. Najťažšie detekovateľný.

Protiopatrenia — 4 fázy reakcie

1
Prevencia: Patch management, access controls, user awareness. NIST: politika · povedomie · mitigácia zraniteľností · mitigácia hrozieb.
2
Detekcia: Zistenie, že infekcia nastala a lokalizácia malwaru.
3
Identifikácia: Určenie konkrétneho malwaru.
4
Odstránenie: Vyčistenie všetkých infikovaných systémov. Ak nie je možné → záloha + rebuild.
Ako prepend parasitický vírus funguje v pamäti

Vírus pri infekcii: (1) prečíta pôvodný súbor do bufferu, (2) vytvorí nový súbor s vlastným kódom na začiatku, (3) za ním pripojí pôvodný obsah. Pôvodný entry point uloží do hlavičky vírusu. Pri spustení: vírus beží → skočí späť na pôvodný entry point → program funguje normálne, používateľ nič nepozoruje.

Moderný malware: Fileless a Ransomware
Fileless malware
Žije výhradne v pamäti RAM, registry alebo WMI subscriptions — žiadne súbory na disku.

Techniky: PowerShell in-memory execution, Process hollowing, DLL injection, Living-Off-the-Land Binaries (LOLBins — zneužitie legitímnych Windows nástrojov ako certutil.exe, mshta.exe).

Prečo ťažko detekovateľný: žiadny súbor → signatúrny antivírus ho nenájde. Potrebné behaviour-based detection.
Ransomware anatómia
  1. Initial access (phishing, exploit)
  2. Lateral movement (šírenie v sieti)
  3. C2 check-in (Command & Control)
  4. Key exchange: vygeneruje AES kľúč, zašifruje ho public kľúčom útočníka → len útočník ho môže dešifrovať
  5. File enumeration (nájde dokumenty, fotky, DB)
  6. Encryption (AES/ChaCha20)
  7. Ransom note + kontakt
Bez zálohy = bez obnovy. Zálohy sú jedinou spoľahlivou obranou.

Otázka 1
Aký je hlavný rozdiel medzi polymorfickým a metamorfickým vírusom?
Polymorfický: mutuje inštrukcie (náhodné no-ops, zmeny kódu) → iný bytový vzor, ale rovnaká logika. Metamorfický: kompletne sa prepíše pri každej infekcii → mení aj správanie, štruktúru kódu → ťažšie detekovateľný aj heuristikou.
Otázka 2
Antivírusový skener 1. generácie funguje na princípe:
1. generácia = signatúrne skenovanie: antivírus pozná 'odtlačky' (hash/bajty) známych vírusov a porovnáva. Výhoda: presné, nula false positives pre známy malware. Nevýhoda: detekuje len známy malware — polymorfický a nový malware prejde.
Otázka 3
Ransomware typicky používa pre šifrovanie súborov obete:
Ransomware hybridné šifrovanie: 1) Vygeneruje náhodný AES kľúč. 2) Zašifruje súbory obete AES (rýchlo). 3) AES kľúč zašifruje verejným kľúčom útočníka (RSA). 4) Bez zaplatenia útočník neprezradí privátny kľúč → obeť nedešifruje AES kľúč.
Otázka 4
Prepend parasitický vírus po infekcii spôsobí, že:
Prepend vírus: (1) Uloží pôvodný entry point. (2) Vloží seba na začiatok súboru. (3) Upraví e_entry na seba. (4) Na konci vírusu JMP na pôvodný kód. Výsledok: vírus beží → šíri sa → potom skočí na pôvodný program → používateľ nič nepozoruje.
Otázka 5
Fileless malware je ťažko detekovateľný, pretože:
Fileless malware = žiadne súbory → signatúrny antivírus nemá čo skenovať. Techniky: PowerShell in-memory, DLL injection, LOLBins (legitímne Windows nástroje). Potrebná behaviour-based detekcia (monitorovanie API volaní, anomálií v procesoch).
KAP 07

Bezpečnosť elektronickej pošty

E-mail je postcardová služba — obsah štandardne nechránený, môže byť prečítaný/modifikovaný v tranzite alebo na cieľovom serveri.

Hrozby e-mailu

Bezpečnosť dát
Sieťová, serverová. Obsah nie je štandardne šifrovaný.
SPAM
Zaberá bandwidth a systémové zdroje.
Phishing
Podvrhnuté e-maily lákajúce na zadanie citlivých údajov.
Malware & Ransomware
Šírenie cez prílohy a klikateľné linky.

Ochranné mechanizmy

SPF — Sender Policy Framework
Štandardizovaný spôsob identifikácie autorizovaných odosielateľov domény. SPF záznamy sú uložené ako TXT DNS záznamy. Overenie prebieha pred prijatím tela správy (from: pole v hlavičke sa nekontroluje).
DKIM — DomainKeys Identified Mail (RFC 4871)
Kryptografické podpisovanie e-mailov. Podpisujúca doména preberá zodpovednosť za správu. Príjemca overuje podpis dotazom na DNS domény odosielateľa — získa verejný kľúč.

PGP — Pretty Good Privacy

Vyvinul Phil Zimmermann. Kombinácia symetrickej a asymetrickej kryptografie + kompresia + Base64 pre e-mailom kompatibilné kódovanie.

SlužbaAlgoritmy
Digitálny podpisSHA-1 + RSA/DSS (autentifikácia, integrita)
Šifrovanie správyCAST-128 / IDEA / 3DES / AES (symetrický, jednorazový session key)
Šifrovanie kľúčaRSA alebo ElGamal — session key sa zašifruje verejným kľúčom príjemcu
KompresiaZIP — pred šifrovaním, po podpise
E-mailová kompatibilitaRadix-64 (Base64) konverzia binárneho streamu na ASCII
PGP Web of Trust: Každý používateľ je CA. Môžeš podpísať kľúče ľudí, ktorých poznáš. Kľúče zdieľané cez keyringy (public-key ring + private-key ring). Private keys šifrované hashom passphrase.

S/MIME

Secure/Multipurpose Internet Mail Extensions — štandardná metóda šifrovania a podpisovania e-mailov. Podporovaný e-mailovými klientmi (Outlook, Thunderbird). Používa X.509 v3 certifikáty — na rozdiel od PGP, používateľ nemôže byť CA.

PGP hybridné šifrovanie — prečo nie len RSA?

RSA je tisícnásobne pomalší ako symetrická šifra. Preto PGP šifruje správu symetricky (rýchlo) a symetrický kľúč asymetricky (bezpečne).

Ako PGP kombinuje symetrickú a asymetrickú kryptografiu pri šifrovaní správy?

Čo overuje SPF (Sender Policy Framework)?

PGP: decentralizovaný Web of Trust — každý môže podpísať kľúče ľudí, ktorých pozná. S/MIME: centralizovaná PKI — X.509 v3 certifikáty vydávajú CA (Certificate Authorities). Používateľ nemôže sám byť CA.

Prečo PGP komprimuje správu (ZIP) PRED šifrovaním, nie po?

Firewally & Systémy prevencie narušenia

Firewall je vložený medzi lokálnu sieť a internet — vytvára kontrolovaný bod vstupu a vonkajší bezpečnostný perimeter (choke point). Realizuje klasickú vojenskú doktrínu "defense in depth".

Ciele návrhu firewallu (Bell94)

  1. Všetka komunikácia dovnútra/von musí prejsť cez firewall.
  2. Len autorizovaná komunikácia (definovaná bezpečnostnou politikou) môže prejsť.
  3. Firewall sám musí byť imúnny voči prieniku — hardened systém.

Útoky na packet filtering firewally

  • IP address spoofing: Falošná zdrojová IP adresa interného hostu → Protiopatrenie: zahodiť pakety z externého rozhrania s internou zdrojovou IP.
  • Source routing attacks: Útočník špecifikuje trasu paketu → Protiopatrenie: zahodiť všetky pakety s touto voľbou.
  • Tiny fragment attacks: Malé fragmenty — TCP hlavička v oddeleným fragmente → Protiopatrenie: prvý fragment musí obsahovať minimálnu veľkosť transport hlavičky.

Výchozí politiky filtra

Default Discard (konzervatívna)
Čo nie je explicitne povolené, je zakázané. Preferovaná firmami a štátom. Spočiatku nepohodlná pre používateľov.
Default Forward (otvorenejšia)
Čo nie je explicitne zakázané, je povolené. Jednoduchšia pre používateľov, ale znižuje bezpečnosť. Typická pre univerzity.

Bastion Host

Kritický bod sieťovej bezpečnosti — hardened systém, platforma pre application/circuit-level gateway. Len nevyhnutné služby (DNS, FTP, HTTP, SMTP proxy). Rozsiahly audit logging.

Obmedzenia firewallov

  • Nechráni pred útokmi, ktoré obídu firewall (modemy, VPN klientov, wireless).
  • Slabá ochrana pred internými hrozbami (nespokojný zamestnanec).
  • Nezabráni prenosným zariadeniam infikovaným mimo siete.
DMZ (Demilitarized Zone) architektúra

Ak útočník kompromituje web server v DMZ, stále nemá priamy prístup do Internal LAN. Inner firewall blokuje priame spojenia z DMZ na DB server — web server komunikuje s DB len cez povolené SQL porty.

KAP 09

Zálohovanie dát (Data Backups)

Záloha vs. Archív
Záloha (Backup): Kópia dát v čase, primárne pre prípad zlyhania aktívnych súborov. Najnovšia záloha sa stáva de facto originálom pri strate aktívnych dát.

Archív: Ukladanie zriedka používaných dát pre referenčné alebo právne účely.

Prečo zálohovať?

  • Nahradiť stratené alebo poškodené dáta
  • Splniť auditné a právne požiadavky
  • Forenzné vyšetrovanie (identifikácia zločinu)
  • Preukázanie autorstva dokumentov

Technológie odolnosti voči zlyhaniam

RAID (Redundant Array of Inexpensive Disks)
RAID 0: Striping — výkon, žiadna redundancia.
RAID 1: Mirroring — presná kópia na 2+ diskoch. Jednoduchá obnova.
RAID 5: Striping + parita (min. 3 disky). Toleruje výpadok 1 disku.
RAID 6: Ako RAID 5 ale 2 paritné bloky — toleruje výpadok 2 diskov.
SAN a NAS
SAN (Storage Area Network): vysokorýchlostná sieť pre blokové úložiská — typicky Fibre Channel. Pre databázy, virtualizáciu.

NAS (Network Attached Storage): súborové úložisko prístupné cez sieť (NFS, SMB/CIFS). Jednoduchá inštalácia, vhodné pre zálohy.

Likvidácia médií

  • Scavenging: Riziko neoprávneného získania dát z vyradených médií.
  • OS formátovanie: Nedostatočné — dáta sú obnoviteľné.
  • Clearing data: Viacnásobný prepis (DoD 5220.22-M).
  • Fyzická deštrukcia (shredding): Najspoľahlivejšie pre citlivé dáta.

RTO, RPO a RAID ≠ Backup
RTO — Recovery Time Objective
Maximálny čas, za ktorý musí byť systém obnovený po výpadku. Definuje ho biznis podľa nákladov výpadku. Príklad: e-shop — RTO = 2 hodiny.
RPO — Recovery Point Objective
Maximálna prípustná strata dát — koľko hodín/dní môžeme stratiť. Určuje frekvenciu zálohovania. Príklad: RPO = 4h → zálohovanie každé 4 hodiny.
RAID ≠ Záloha! RAID-1 (zrkadlenie) chráni pred HDD zlyhaním, ALE:
• Ak zmažeš súbor → zmazanie sa ihneď zrkadlí
• Ransomware zašifruje súbory → RAID zašifruje obe kópie zároveň
• RAID nechráni pred: logickými chybami, vírusmi, náhodným vymazaním, požiarom

RAID je HA (High Availability) riešenie. Záloha je DR (Disaster Recovery) riešenie. Oboje je potrebné.

Moderné rozšírenie: aspoň jedna kópia nemeniteľná (WORM storage, S3 Object Lock) — ransomware ju nemôže zašifrovať ani vymazať.

$ setsebool httpd_can_network_connect on  # dočasne
$ setsebool -P httpd_can_network_connect on  # trvalé

# Logy SELinux
$ audit2why < /var/log/audit/audit.log

Vytvorenie používateľov a rolí

# Zobrazenie SELinux používateľov
$ semanage user -l

# Priradenie linux používateľa k SELinux roli
$ semanage login -a -s staff_u jozef

# Roly v SELinux
# user_u  → bežný používateľ, bez sudo
# staff_u → môže používať sudo pre povolené operácie
# sysadm_u → plný administrátorský prístup
SELinux Label formát — user:role:type:level
system_u:object_r:httpd_sys_content_t:s0
#  │         │          │                 │
#  │         │          │                 └── MLS/MCS bezp. úroveň
#  │         │          └──────────────────── typ (najdôležitejší!)
#  │         └─────────────────────────────── rola
#  └───────────────────────────────────────── SELinux používateľ

Type Enforcement (TE) je jadro SELinuxu — pravidlo hovorí "httpd_t môže čítať httpd_sys_content_t". Ak web server (httpd_t) skúsi čítať súbor s iným typom (napr. shadow_t), SELinux to zablokuje.

Čítanie AVC denial záznamu
type=AVC msg=audit(1234567890.123:456): avc: denied
  { read } for pid=1234 comm="httpd"
  name="secret.txt" dev="sda1" ino=67890
  scontext=system_u:system_r:httpd_t:s0
  tcontext=unconfined_u:object_r:user_home_t:s0
  tclass=file permissive=0
Čo sa stalo
httpd (PID 1234) sa pokúsil čítať súbor secret.txt — a bol zablokovaný.
Prečo zablokované
Zdrojový kontext httpd_t nemá povolenie čítať cieľový kontext user_home_t. Fix: chcon alebo restorecon.

Vírusy — Parazitický vírus v C

🦠

Cieľ

Naprogramovať prepend parasitický vírus v C. Spustiť VÝHRADNE v izolovanom VM (Host-Only sieť)!

VAROVANIE: Toto cvičenie musí prebiehať VÝHRADNE v izolovanej VM (bez sieťového pripojenia alebo Host-Only). Nikdy nespúšťajte vírusový kód na reálnom systéme!

Štruktúra parasitického vírusu

/* Zjednodušená štruktúra prepend vírusu */
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>

#define VIRUS_SIZE 512  /* veľkosť vírusového kódu */

void infect(const char *filename) {
    FILE *target, *tmp;
    char buf[4096];
    size_t n;

    /* 1. Otvoriť cieľový súbor */
    target = fopen(filename, "rb");
    if (!target) return;

    /* 2. Skopírovať pôvodný obsah */
    tmp = fopen("/tmp/.vir_tmp", "wb");
    while ((n = fread(buf, 1, sizeof(buf), target)) > 0)
        fwrite(buf, 1, n, tmp);
    fclose(target);
    fclose(tmp);

    /* 3. Prepísať súbor: vírus + pôvodný obsah */
    target = fopen(filename, "wb");
    /* ... zapísať kód vírusu ... */
    /* ... pripojiť pôvodný obsah z /tmp/.vir_tmp ... */
    fclose(target);
}

int main(int argc, char *argv[]) {
    /* Payload — demonstračný */
    printf("[VIRUS] Infikovaný systém!\n");

    /* Hľadanie a infikovanie súborov */
    /* ... skenovanie adresára ... */

    /* Spustenie pôvodného programu */
    /* execv(original_entry_point, ...) */
    return 0;
}

Kompilacia a testovanie

# Kompilácia (iba v izolovanej VM!)
$ gcc -o virus virus.c

# Vytvorenie testovacích "obetí"
$ echo '#!/bin/bash\necho "Hello World"' > target1
$ echo '#!/bin/bash\necho "Program 2"' > target2
$ chmod +x target1 target2

# Spustenie vírusu
$ ./virus target1 target2

# Overenie infekcie
$ file target1
$ hexdump -C target1 | head -20
ELF (Executable and Linkable Format) štruktúra

Prepend vírus: (1) prečíta pôvodný ELF header a uloží pôvodnú hodnotu e_entry (entry point), (2) zapíše seba na začiatok súboru, (3) upraví e_entry aby ukazoval na vírusový kód, (4) na konci vírusu JMP na pôvodnú e_entry. Keď OS načíta ELF, skočí na vírus, nie na pôvodný kód.

Bezpečnosť e-mailu (PGP / GPG)

✉️

Cieľ

Vygenerovať PGP kľúčový pár, podpísať a zašifrovať e-mail cez GPG.

GPG — základné príkazy

# Generovanie kľúčového páru
$ gpg --gen-key
# Zadaj: meno, e-mail, passphrase (silné!)

# Zobrazenie kľúčov
$ gpg --list-keys          # verejné kľúče
$ gpg --list-secret-keys   # súkromné kľúče

# Export verejného kľúča (na zdieľanie)
$ gpg --armor --export vas@email.sk > public_key.asc

# Import cudzieho verejného kľúča
$ gpg --import kolega_public.asc

# Podpísanie a šifrovanie správy
$ gpg --sign --encrypt --armor \
      --recipient kolega@email.sk \
      -o sprava.gpg \
      sprava.txt

# Dešifrovanie a overenie podpisu
$ gpg --decrypt sprava.gpg

# Len podpisanie (bez šifrovania)
$ gpg --clearsign sprava.txt
# Výstup: sprava.txt.asc s podpisom v čitateľnej forme

# Overenie podpisu
$ gpg --verify sprava.txt.asc

Odovzdanie kľúčov

# Nahranie verejného kľúča na keyserver
$ gpg --keyserver keys.openpgp.org --send-keys KEYID

# Stiahnutie kľúča kolegu
$ gpg --keyserver keys.openpgp.org --recv-keys KEYID

# Podpisovanie cudzieho kľúča (web of trust)
$ gpg --sign-key kolega@email.sk

Zálohovanie — rsync & Duplicati

💾

Cieľ

Nakonfigurovať automatické zálohovanie pomocou rsync a Duplicati.

rsync — zálohovanie príkazovým riadkom

# Základná syntax
rsync [OPTIONS] SOURCE DESTINATION

# Lokálna záloha s verbóze výstupom
$ rsync -avz /home/user/ /backup/user/
# -a = archive (zachová oprávnenia, dátumy, symlinky)
# -v = verbose
# -z = kompresia

# Inkrementálna záloha s hardlinks (šetrí miesto)
$ rsync -avz --link-dest=/backup/yesterday/ \
        /home/user/ /backup/today/

# Záloha cez SSH na vzdialený server
$ rsync -avz -e ssh /home/user/ backup@server:/backup/

# Záloha s vylúčením adresárov
$ rsync -avz --exclude='*.tmp' \
        --exclude='node_modules/' \
        /home/user/ /backup/user/

# Automatická záloha cez cron (každú noc o 2:00)
$ crontab -e
0 2 * * * rsync -az /home/ /backup/home/ >> /var/log/backup.log 2>&1

Duplicati — GUI zálohovanie s šifrovaním

1
Stiahnuť Duplicati z duplicati.com → nainštalovať
2

3
Pridať zálohu → vybrať zdroj (adresáre na zálohovanie)

5
Nastaviť šifrovanie (AES-256) — zapísať si passphrase!
6
Nastaviť plán (napr. denná záloha o 3:00)

Otestovať obnovu! Záloha bez testovanej obnovy je bezcenná.
